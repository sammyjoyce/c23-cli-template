## Review

- Blocker: `docs/ZIG_PRIMER.md` and `CONTRIBUTING.md` now direct users to Zig 0.16.0, but their build-system snippets still use pre-0.16 `Step.Compile` APIs. This makes the documentation portion of the 0.16 migration internally inconsistent even though the actual `build.zig` source was migrated.
  - Evidence: `docs/ZIG_PRIMER.md:63-67` constructs an executable with `.target` / `.optimize` fields directly on `b.addExecutable`, while Zig 0.16.0's `std.Build.ExecutableOptions` accepts `.name` and `.root_module` instead; the migrated source uses the 0.16 shape at `build.zig:40-47`.
  - Evidence: `docs/ZIG_PRIMER.md:70`, `docs/ZIG_PRIMER.md:86-90`, `docs/ZIG_PRIMER.md:109-110`, `docs/ZIG_PRIMER.md:198`, `docs/ZIG_PRIMER.md:251-255`, `docs/ZIG_PRIMER.md:273-281`, `docs/ZIG_PRIMER.md:329`, `docs/ZIG_PRIMER.md:342`, `docs/ZIG_PRIMER.md:357`, and `CONTRIBUTING.md:178` use `exe.addCSourceFiles`, `exe.addCSourceFile`, `exe.addIncludePath`, or `exe.linkSystemLibrary`; in Zig 0.16 these calls belong on `exe.root_module` / `std.Build.Module`, matching the migrated source at `build.zig:49`, `build.zig:88-90`, `build.zig:113-115`, and `build.zig:120-134`.
  - Fix needed: update the docs/contributing snippets to the Zig 0.16 `root_module = b.createModule(...)` / `exe.root_module.addCSourceFile(s)` / `exe.root_module.addIncludePath` / `exe.root_module.linkSystemLibrary(..., .{})` forms, or replace them with references to the actual migrated `build.zig`.
